# codemap v0.2 Implementation Plan

Implements the spec in the v0.2 design doc. Assumes v0.1 is complete: index, map, query, and serve commands all work. The codebase is flat (`src/*.rs`) with no `commands/` or `lib/` subdirectories.

---

## Overview

Nine phases, ordered by dependency. Phases 3–5 are independent and can be done in parallel.

```
Phase 1: Remove MCP server
  ↓
Phase 2: DB helpers (new queries needed by later phases)
  ↓
Phase 3: Symbol command  ─┐
Phase 4: Deps command     ├─ independent of each other
Phase 5: Context command ─┘
  ↓
Phase 6: Map output restructure (references new commands in footer)
  ↓
Phase 7: Incremental index
  ↓
Phase 8: Setup command (depends on map + incremental existing)
  ↓
Phase 9: Directory restructure (optional, do last)
```

## Phase 2: DB Helper Queries

**Goal:** Add database functions needed by Phases 3–8.

### Steps

1. **Add `get_stats(conn) -> Result<(usize, usize, usize)>`** to `src/db.rs`.
   Returns `(file_count, exported_symbol_count, edge_count)`. Three `SELECT COUNT(*)` queries. The symbol count filters to `is_exported = 1` only (matches the "847 exports" phrasing in the spec).
   ```rust
   pub fn get_stats(conn: &Connection) -> Result<(usize, usize, usize)> {
       let files: usize = conn.query_row("SELECT COUNT(*) FROM files", [], |r| r.get(0))?;
       let exports: usize = conn.query_row(
           "SELECT COUNT(*) FROM symbols WHERE is_exported = 1", [], |r| r.get(0)
       )?;
       let edges: usize = conn.query_row("SELECT COUNT(*) FROM edges", [], |r| r.get(0))?;
       Ok((files, exports, edges))
   }
   ```

2. **Add `get_importer_counts(conn) -> Result<HashMap<i64, usize>>`** to `src/db.rs`.
   Batch query: `SELECT target_id, COUNT(*) FROM edges GROUP BY target_id`.
   Returns file_id → number of files that import it. Used by the map command for `[12 importers]` metadata.

3. **Add `get_all_files_with_exports(conn) -> Result<Vec<FileWithExports>>`** to `src/db.rs`.
   New struct:
   ```rust
   pub struct FileWithExports {
       pub path: String,
       pub rank: f64,
       pub exports: Vec<String>,  // exported symbol names
   }
   ```
   Query joins `files` with `symbols WHERE is_exported = 1`, groups by file. Used by the context command's scoring engine.

4. **Add `mtime` column to `files` table** in `init_db()`.
   Append to the `CREATE TABLE IF NOT EXISTS files` block:
   ```sql
   -- After the existing CREATE TABLE, add:
   ALTER TABLE files ADD COLUMN mtime INTEGER;
   ```
   Since `ALTER TABLE ... ADD COLUMN` fails if the column already exists, wrap in a check: query `PRAGMA table_info(files)` first, only ALTER if `mtime` is absent. Alternatively, use a simpler approach: attempt the ALTER and swallow the "duplicate column" error.

5. **Add `get_file_mtime(conn, path) -> Option<i64>`** and **`update_file_mtime(conn, path, mtime)`** to `src/db.rs`. Used by incremental indexing in Phase 7.

### Files touched
- `src/db.rs` — add ~60 lines (4 functions + 1 struct + schema migration)

### Verify
- Existing `codemap index` + `codemap map` still work (schema change is additive).
- New functions return correct values against a real indexed project.

## Phase 4: Add `codemap deps` Command

**Goal:** Show the import graph neighborhood of a file.

### Steps

1. **Create `src/deps.rs`** with `run_deps(root, file_path, direction, depth, all, json)`.

2. **Core logic:**
   - Normalize input path: strip `root` prefix if absolute, convert to forward slashes.
   - Open DB, verify file exists in index (bail with helpful message if not).
   - Call `db::get_file_deps(conn, path, "imports")` and/or `db::get_file_deps(conn, path, "importers")` based on `--direction`.
   - Default direction is `both`.

3. **Multi-hop traversal (`--depth N`):**
   - BFS from the target file.
   - At each hop, get deps of the current frontier.
   - Track visited set to avoid cycles.
   - Collect results with hop distance metadata.
   - Default depth is 1.

4. **Importer truncation:**
   - Show max 5 importers by default.
   - `--all` shows every importer.

5. **Output format:**
   ```
   src/services/auth.ts

   Imports (3):
     src/db/users.ts         [Import]
     src/utils/crypto.ts     [Import]
     src/config/jwt.ts       [Import]

   Imported by (12):
     src/routes/auth.routes.ts
     src/routes/api.ts
     src/middleware/requireAuth.ts
     src/hooks/useAuth.ts
     src/services/admin.ts
     ... and 7 more (use --all to show all)
   ```

   `--json`: structured JSON with `{file, imports: [{path, edge_type}], importers: [{path, edge_type}]}`.

6. **Edit `src/main.rs`:** Add `Commands::Deps` variant:
   ```rust

       /// Show all importers without truncation
   ```

7. **Edit `src/lib.rs`:** Add `pub mod deps;`.

## Phase 5: Add `codemap context` Command

**Goal:** Keyword-based file relevance scoring. No LLM, purely local.

### Steps

1. **Create `src/scorer.rs`** — the keyword scoring engine:

   **`tokenize_query(query: &str) -> Vec<String>`:**
   - Split on whitespace.
   - Lowercase everything.
   - Remove stop words: `a, an, the, is, are, was, were, be, been, being, to, for, and, or, of, in, on, with, at, by, from, as, it, its, this, that`.
   - Return deduplicated keyword list.

   **`score_files(keywords, files: &[FileWithExports], conn) -> Vec<ScoredFile>`:**
   - For each file, compute a raw score:
     - **Filename match:** For each keyword, if any path component (split on `/` and `.`) contains the keyword → +3.0.
     - **Export name match:** For each keyword, if any exported symbol name (lowercased) contains the keyword → +2.0.
   - Multiply raw score by the file's PageRank score.
   - Filter to files with score > 0.
   - **Graph expansion:** For each scored file, look up its 1-hop neighbors (imports + importers) via `db::get_file_deps`. Add neighbors at 0.3× the parent's score. If a neighbor already has a score, take the max.
   - Sort descending by combined score.

   **`ScoredFile` struct:**
   ```rust
   pub struct ScoredFile {
       pub path: String,
       pub score: f64,
       pub rank: f64,
       pub match_reasons: Vec<String>,  // e.g., ["auth", "login"]
   }
   ```

2. **Create `src/context.rs`** with `run_context(root, query, limit, json, include_content)`:
   - Load all files with exports via `db::get_all_files_with_exports()`.
   - Run `scorer::score_files(...)`.
   - Take top N (default 10).
   - Print results.

   **Output format:**
   ```
   Relevant files for: "add OAuth Google login support"

   src/services/auth.ts        [rank: 0.94 | match: auth, login]
   src/routes/auth.routes.ts   [rank: 0.68 | match: auth]
   src/middleware/session.ts    [rank: 0.55 | match: session → auth]
   src/config/oauth.ts         [rank: 0.42 | match: oauth]
   src/models/user.ts          [rank: 0.82 | match: user → auth]
   ```

   Files that scored via graph expansion (not direct keyword match) show the connection with `→`: `session → auth` means "session.ts imports auth.ts which matched directly."

   `--include-content`: after the ranked list, print full file contents for each matched file (read from disk). Useful for piping into prompts.

   `--json`: structured output with `{query, results: [{path, score, rank, match_reasons}]}`.

3. **Edit `src/main.rs`:** Add `Commands::Context` variant:
   ```rust
   /// Suggest the most relevant files for a task
   Context {
       /// Natural language task description
       #[arg()]
       query: String,

       /// Max results
       #[arg(short, long, default_value_t = 10)]
       limit: usize,

       /// JSON output
       #[arg(long)]
       json: bool,

       /// Also print file contents
       #[arg(long)]
       include_content: bool,
   },
   ```

4. **Edit `src/lib.rs`:** Add `pub mod scorer; pub mod context;`.

### Files touched
- `src/scorer.rs` — new (~100 lines)
- `src/context.rs` — new (~80 lines)
- `src/main.rs` — add command variant + match arm
- `src/lib.rs` — add 2 modules

### Verify
- `codemap context "add authentication"` returns ranked files with auth-related paths first.
- `codemap context "add authentication" --json` outputs valid JSON.
- `codemap context "xyzzy gibberish"` returns empty result (no matches).
- Files reached via graph expansion show the `→` connection notation.


   Change the file header line from:
   ```

   ```
   to:
   ```

   ```
   Call `db::get_importer_counts()` once, look up each file's count. Format rank to 2 decimal places.

   ```

   Run these in Bash for deeper codebase queries:
   - `codemap deps <file>` — imports and importers of a file

   - `codemap context "<task>"` — suggest the most relevant files for a task

   ```

   - Code map body: remaining budget
   - So with `--tokens 1500` and instructions on: body gets ~1390 tokens.

## Phase 7: Add `--incremental` to `codemap index`

**Goal:** Fast re-indexing (<500ms for a single changed file) for the PostToolUse hook.

### Steps

1. **Ensure `mtime` column exists** (done in Phase 2).

2. **Add `--incremental` flag to `IndexArgs` in `src/main.rs`:**
   ```rust
   #[derive(Args, Debug)]
   struct IndexArgs {
       /// Force full re-index (ignore cache)
       #[arg(long)]
       force: bool,

       /// Incremental: only re-index files with newer mtime
       #[arg(long)]
       incremental: bool,
   }
   ```

3. **Update `run_index()` signature** in `src/index.rs`:
   ```rust
   pub fn run_index(root: &Path, force: bool, incremental: bool) -> Result<()>
   ```

4. **Incremental change detection** (new logic in `run_index`, before the existing hash loop):

   When `incremental` is true:
   ```
   for each discovered file:
     stat() the file → get fs_mtime
     db_mtime = db::get_file_mtime(conn, rel_path)
     if db_mtime is Some and fs_mtime <= db_mtime:
       skip (don't even read the file)
     else:
       read file, hash, compare hash as before
       if hash unchanged: update mtime only, skip parse
       if hash changed: mark for re-parse
   ```

   When `incremental` is false (default): existing behavior. Read every file, hash, compare hash.

   When `force` is true: ignore both mtime and hash. Reparse everything.

5. **Early exit optimization:**
   When incremental and zero files changed (all mtimes matched), skip graph construction and PageRank entirely. Print "codemap: up to date (0 changes)" and return.

6. **Store mtime after successful indexing:**
   After upserting a file to the DB, also store its mtime:
   ```rust
   let metadata = std::fs::metadata(&abs_path)?;
   let mtime = metadata.modified()?.duration_since(UNIX_EPOCH)?.as_secs() as i64;
   db::update_file_mtime(&tx, &rel_path, mtime)?;
   ```

7. **PageRank still runs on the full graph** even in incremental mode (when there are changes). PageRank is a global algorithm — changing one file's edges can affect ranks of all files. For 300 files, PageRank takes ~5ms, so this is fine.

8. **Update the `run_index` call in `src/main.rs`** to pass the new flag.

### Files touched
- `src/index.rs` — add incremental path (~40 lines)
- `src/main.rs` — add flag, update call
- `src/db.rs` — (already done in Phase 2)

### Verify
- `codemap index` (full) works as before.
- `codemap index --incremental` on a clean project: "up to date (0 changes)", completes in <100ms.
- Touch one file, `codemap index --incremental`: re-indexes only that file, completes in <500ms.
- `codemap index --force` always re-parses everything.
- `--force` and `--incremental` together: `--force` wins (re-parse all).

## Phase 8: Add `codemap setup` Command

**Goal:** Single command that configures everything. No CLAUDE.md, no MCP, no manual hook editing.

### Steps

1. **Create `src/setup.rs`** with `run_setup(root, no_post_hook, global, dry_run)`.

2. **Step A — Run index:**
   ```rust
   eprintln!("Indexing codebase...");
   crate::index::run_index(root, false, false)?;
   let conn = db::init_db(db_path_str)?;
   let (files, exports, edges) = db::get_stats(&conn)?;
   eprintln!("✓ Indexed {files} files ({exports} exports, {edges} edges)");
   ```

3. **Step B — Determine config path:**
   ```rust
   let config_path = if global {
       dirs::home_dir().unwrap().join(".claude/settings.json")
   } else {
       root.join(".claude/settings.local.json")
   };
   ```
   Note: avoid adding a `dirs` dependency. Use `std::env::var("HOME")` on Unix or `USERPROFILE` on Windows:
   ```rust
   let home = std::env::var("HOME")
       .or_else(|_| std::env::var("USERPROFILE"))
       .map(PathBuf::from)?;
   ```

4. **Step C — Read existing config:**
   ```rust
   let mut config: serde_json::Value = if config_path.exists() {
       let content = std::fs::read_to_string(&config_path)?;
       serde_json::from_str(&content)?
   } else {
       serde_json::json!({})
   };
   ```

5. **Step D — Build hook entries:**
   ```rust
   let session_hook = serde_json::json!({
       "hooks": [{
           "type": "command",
           "command": "codemap map",
           "timeout": 10
       }]
   });

   let post_tool_hook = serde_json::json!({
       "matcher": "Write|Edit|MultiEdit",
       "hooks": [{
           "type": "command",
           "command": "codemap index --incremental",
           "timeout": 30,
           "async": true
       }]
   });
   ```

6. **Step E — Merge into config:**
   - Ensure `config["hooks"]` is an object. Create if missing.
   - For `SessionStart`: get `config["hooks"]["SessionStart"]` as array. Remove any existing entry whose `hooks[].command` starts with `"codemap"`. Append the new entry.
   - For `PostToolUse` (unless `--no-post-hook`): same merge logic — remove existing codemap entries, append new.
   - This merge strategy preserves hooks from other tools.

7. **Step F — Write config:**
   ```rust
   if dry_run {
       eprintln!("Would write to {}:", config_path.display());
       eprintln!("{}", serde_json::to_string_pretty(&config)?);
   } else {
       std::fs::create_dir_all(config_path.parent().unwrap())?;
       std::fs::write(&config_path, serde_json::to_string_pretty(&config)?)?;
   }
   ```

8. **Step G — Print summary:**
   ```
   ✓ Indexed 142 files (847 exports, 312 edges)
   ✓ Wrote SessionStart hook to .claude/settings.local.json
   ✓ Wrote PostToolUse hook (async re-index on file changes)

   codemap is ready. Start Claude Code and it will automatically
   receive the codebase map and know how to use codemap commands.
   ```

9. **Edit `src/main.rs`:** Add `Commands::Setup` variant:
   ```rust
   /// Configure Claude Code hooks (the only setup step needed)
   Setup {
       /// Skip the PostToolUse re-indexing hook
       #[arg(long)]
       no_post_hook: bool,

       /// Write to ~/.claude/settings.json instead of project-local
       #[arg(long)]
       global: bool,

       /// Print what would be written without writing
       #[arg(long)]
       dry_run: bool,
   },
   ```

10. **Edit `src/lib.rs`:** Add `pub mod setup;`.

### Files touched
- `src/setup.rs` — new (~130 lines)
- `src/main.rs` — add command variant + match arm
- `src/lib.rs` — add module

### Verify
- `codemap setup --dry-run` prints the config without writing.
- `codemap setup` creates `.claude/settings.local.json` with correct hook config.
- Running `codemap setup` again doesn't duplicate hooks (merge logic replaces).
- `codemap setup --no-post-hook` omits the PostToolUse entry.
- `codemap setup --global` writes to `~/.claude/settings.json`.
- Existing non-codemap hooks in the config file are preserved.

   The spec suggests a `src/lib/` subdirectory but this would conflict with Rust's `lib.rs` convention for library crates. Keeping library modules at `src/` root and command modules under `src/commands/` is more idiomatic.

3. **Create `src/commands/mod.rs`:**
   ```rust
   pub mod index;
   pub mod map;
   pub mod symbol;
   pub mod deps;
   pub mod context;
   pub mod setup;
   ```

4. **Update `src/lib.rs`:**
   ```rust
   pub mod commands;
   pub mod store;  // renamed from db
   pub mod graph;
   pub mod hash;
   pub mod parser;
   pub mod resolver;
   pub mod scorer;
   pub mod types;
   pub mod walk;
   ```

5. **Update all `use crate::` imports** across every file. The main change: `crate::db` → `crate::store`, and command modules move from `crate::index` to `crate::commands::index`.

6. **Update `src/main.rs`** call sites: `codemap::index::run_index` → `codemap::commands::index::run_index`, etc.

### Files touched
- Every `.rs` file (import path updates)

### Risk
High churn, many files. Do this atomically in a single commit. Run `cargo test` and `cargo build` before committing.

## Final CLI Surface

After all phases:

```
codemap index [--force] [--incremental]      Build or update the code map index
codemap map [--tokens N] [--no-instructions]  Print ranked code map to stdout
codemap deps <file> [--depth N] [--all]       Show imports/importers of a file
codemap symbol <pattern> [--all] [--exact]    Find a symbol across the codebase
codemap context "<query>" [--limit N]         Suggest relevant files for a task
codemap setup [--no-post-hook] [--dry-run]    Configure Claude Code hooks
```

No `serve`. No `query` (replaced by `symbol`). No MCP. No CLAUDE.md edits needed.
